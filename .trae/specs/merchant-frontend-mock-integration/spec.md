# 商家端前端Mock集成 - 产品需求文档

## 概述
- **摘要**：修改前端项目中商家端相关页面的代码，将其调整为使用mock模拟后端接口和测试数据，确保在排除后端服务依赖的情况下独立验证前端功能。
- **目的**：使前端开发能够在没有真实后端的情况下进行页面开发和测试，验证前端页面的功能完整性、交互逻辑正确性和UI展示效果。
- **目标用户**：前端开发人员、测试人员

## 目标
- 将商家端所有页面的API调用调整为使用mock服务
- 配置适当的mock数据生成规则，确保数据结构与真实后端接口一致
- 提供便捷的mock数据管理方式
- 确保前端页面在使用mock数据时功能完整、交互逻辑正确、UI展示效果良好

## 非目标（超出范围）
- 开发新的mock服务（假设已有mock服务可用）
- 修改后端服务代码
- 实现真实的业务逻辑
- 处理复杂的业务流程和事务

## 背景与上下文
- 前端项目使用Vue 3 + TypeScript + Element Plus
- 后端项目使用Spring Boot 3.2.0 + Spring Data JPA
- 商家端页面包括：布局组件、注册页面、首页、服务管理、商品管理、订单管理、评价管理、店铺管理、统计报表等
- 已有mock服务基础设施（基于mock.js）

## 功能需求
- **FR-1**：修改商家端所有页面的API调用，使其使用mock服务
- **FR-2**：配置mock数据生成规则，确保数据结构与真实后端接口一致
- **FR-3**：提供便捷的mock数据管理方式，便于测试和调试
- **FR-4**：确保所有页面在使用mock数据时功能完整、交互逻辑正确
- **FR-5**：验证前端页面在不同数据场景下的展示效果

## 非功能需求
- **NFR-1**：mock数据结构与真实后端API响应结构保持一致
- **NFR-2**：修改后的代码结构清晰，易于维护和扩展
- **NFR-3**：页面加载和交互响应时间控制在合理范围内
- **NFR-4**：代码修改遵循现有的代码风格和规范

## 约束
- **技术**：使用现有的mock服务基础设施
- **依赖**：依赖mock服务能够正常运行
- **文件结构**：保持现有的前端项目文件结构

## 假设
- 前端项目已经完成基本结构搭建
- mock服务已经配置完成并能够正常运行
- 后端API接口设计已经确定

## 验收标准

### AC-1：API调用调整
- **给定**：商家端页面代码
- **当**：修改API调用为使用mock服务时
- **则**：所有API调用都通过mock服务处理，不再调用真实后端接口
- **验证**：`programmatic`

### AC-2：mock数据配置
- **给定**：mock服务和前端页面需求
- **当**：配置mock数据生成规则时
- **则**：生成的数据结构与真实后端接口一致，能够满足前端页面的渲染和交互需求
- **验证**：`human-judgment`

### AC-3：功能完整性
- **给定**：修改后的前端页面
- **当**：使用mock数据时
- **则**：页面功能完整，交互逻辑正确，UI展示效果良好
- **验证**：`human-judgment`

### AC-4：数据管理便捷性
- **给定**：mock数据管理方式
- **当**：进行测试和调试时
- **则**：能够便捷地管理和修改mock数据
- **验证**：`human-judgment`
